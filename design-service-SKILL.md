# 스킬: 서비스 아키텍처 설계

## 설명

Analyzer의 분석 결과를 바탕으로 구체적인 개선 계획을 수립합니다.
- 마이크로서비스 분리 계획
- 페이즈별 구현 로드맵
- 위험도 평가 및 롤백 전략
- 배포 전략 수립

## 사용 방법

```bash
# 기본 사용
/design-service

# Analyzer 결과를 바탕으로 설계
/design-service --based-on analyzer-output

# 특정 페이즈만 설계
/design-service --phase 1
/design-service --phase 2
/design-service --phase 3

# 상세 설계 (롤백 계획, 배포 전략 포함)
/design-service --detailed
```

## 설계 프로세스

### 1단계: 분석 결과 검토

Analyzer로부터 받은 JSON을 분석:
- N+1 쿼리 12개 발견
- 테스트 커버리지 67%
- 아키텍처: 모놀리식

### 2단계: 목표 설정

```
현재 상태:
- API 응답시간: 850ms (avg), 1200ms (p95)
- DB 쿼리: 45개/요청
- 테스트 커버리지: 67%

목표 상태:
- API 응답시간: 400ms (avg), 650ms (p95) - 53% 개선
- DB 쿼리: 8개/요청 - 82% 감소
- 테스트 커버리지: 95%
```

### 3단계: 페이즈 분류

개선 작업을 3가지 페이즈로 구분:
- **Phase 1 (Quick Wins)**: 1주, 낮은 위험
- **Phase 2 (Architecture)**: 2주, 중간 위험
- **Phase 3 (Optimization)**: 1주, 낮은 위험

### 4단계: 각 페이즈별 작업 정의

각 페이즈마다:
- 구체적인 작업 목록
- 영향받는 파일
- 예상 시간
- 기술적 난이도
- 예상 성능 개선
- 롤백 계획

### 5단계: 위험 평가

각 페이즈마다:
- 위험도 레벨 (낮음/중간/높음)
- 예상 이슈
- 모니터링 항목
- 자동 롤백 조건

### 6단계: 배포 전략 수립

Phase 2 (Architecture)의 경우:
```
카나리 배포:
Hour 1: 10% 트래픽 → 새 Auth Service
Hour 2: 25% 트래픽 (정상이면 진행)
Hour 3: 50% 트래픽
Hour 4: 100% 트래픽 (완료)

이상 발생 시:
→ 즉시 100% → 이전 버전 롤백 (Load balancer 설정 변경, 2분)
```

## Phase 1: Quick Wins (Week 1) [낮음 위험]

### 목표
빠른 성능 개선과 기초 안정화

### 작업 목록

#### 1.1 DataLoader 도입
- **목표**: N+1 쿼리 제거
- **파일**: src/api/users.ts, src/api/posts.ts
- **변경사항**: 
  - src/loaders/userLoader.ts (신규)
  - src/loaders/postLoader.ts (신규)
  - src/api/users.ts (수정)
  - src/api/posts.ts (수정)
- **시간**: 2일
- **난이도**: 낮음
- **성능 개선**: 850ms → 750ms (12%)
- **테스트**: 새 테스트 8개 추가

#### 1.2 DB 인덱스 추가
- **목표**: 쿼리 성능 개선
- **테이블**: users, posts, comments
- **인덱스 추가**:
  - users(email) - 로그인 최적화
  - posts(user_id) - 사용자별 게시물
  - comments(post_id) - 게시물별 댓글
- **시간**: 1일
- **난이도**: 낮음
- **성능 개선**: 750ms → 650ms (13%)
- **롤백**: npx migrate:rollback (1분)

#### 1.3 테스트 추가 (Auth 서비스)
- **목표**: 커버리지 67% → 78%
- **파일**: src/services/auth.ts
- **테스트 케이스**: 15개
  - 로그인 성공
  - 로그인 실패
  - 토큰 갱신
  - 에러 케이스 등
- **시간**: 2일
- **난이도**: 낮음

### Phase 1 완료 후
- **API 응답시간**: 850ms → 650ms (24% 개선)
- **DB 쿼리**: 45 → 35개 (22% 감소)
- **테스트 커버리지**: 67% → 78%
- **위험도**: 낮음 (모두 롤백 가능)

---

## Phase 2: Architecture Split (Week 2-3) [중간 위험]

### 목표
마이크로서비스 기초 구성, 확장성 개선

### 작업 목록

#### 2.1 Auth Service 분리
- **목표**: Auth 로직을 별도 서비스로 분리
- **현재**: src/services/auth.ts (모놀리식 내부)
- **변경**: 독립 서비스 (포트 3001)
- **통신**: REST API
- **파일 변경**:
  - src/services/auth.ts → 제거
  - services/auth/ 디렉토리 (신규)
  - src/api/auth-routes.ts (업데이트, API Gateway)
- **데이터베이스**: 공유 (향후 분리 예정)
- **시간**: 3일
- **난이도**: 중간
- **성능 개선**: 직접적 개선 없음 (향후 최적화 기반)
- **롤백**: Git revert (5분) + API Gateway 설정 변경
- **테스트**: 통합 테스트 10개 추가

#### 2.2 API Gateway 구성
- **목표**: 요청 라우팅, 인증 통합
- **도구**: Express + http-proxy
- **포트**: 3000 (기존), Auth Service (3001)
- **라우팅 규칙**:
  - /auth/* → Auth Service (3001)
  - /* → API Service (3000)
- **시간**: 2일
- **난이도**: 낮음
- **테스트**: E2E 테스트 5개

#### 2.3 카나리 배포 구성
- **전략**: 10% → 25% → 50% → 100%
- **각 단계**: 1시간 모니터링
- **롤백 트리거**: 에러율 > 1% or 응답시간 > 1000ms
- **자동 롤백**: Load balancer 설정 변경 (1분)
- **시간**: 1일 (인프라 구성)

### Phase 2 완료 후
- **API 응답시간**: 650ms → 550ms (15% 추가 개선, 누적 35%)
- **아키텍처**: 모놀리식 → 서비스 분리 기초 완성
- **다음 확장성**: Auth Service 독립 배포 가능

---

## Phase 3: Optimization (Week 4) [낮음 위험]

### 목표
최종 성능 최적화, 캐싱 도입

### 작업 목록

#### 3.1 Redis 캐싱 추가
- **목표**: 반복 조회 최적화
- **도구**: Redis
- **캐시 대상**: User 조회 (예상 80% 캐시율)
- **TTL**: 1시간
- **무효화 전략**: 사용자 수정 시 캐시 제거
- **파일 변경**:
  - src/cache/userCache.ts (신규)
  - src/loaders/userLoader.ts (업데이트, 캐시 활용)
- **시간**: 2일
- **난이도**: 낮음
- **성능 개선**: 550ms → 450ms (18%)
- **테스트**: 캐시 테스트 8개

#### 3.2 나머지 쿼리 최적화
- **목표**: 개별 쿼리 성능 미세 조정
- **예상**: 8-10개 추가 최적화 발견 및 구현
- **시간**: 2일
- **성능 개선**: 450ms → 400ms (11%)

#### 3.3 부하 테스트
- **도구**: Apache Bench (ab) 또는 k6
- **목표**: 1000 RPS @ p95 400ms 이하
- **시간**: 1일
- **결과**: 최종 성능 검증

### Phase 3 완료 후
- **API 응답시간**: 550ms → 400ms (27% 추가 개선, 누적 53%)
- **DB 쿼리**: 35 → 8개 (77% 추가 감소, 누적 82%)
- **테스트 커버리지**: 95% 달성
